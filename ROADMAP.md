# Extended Properties — state of the tool, gaps, and forward architecture

An assessment of the plugin as of v2.10.4: what exists, what is missing, how missing pieces should be built, and what constraints should shape future work.

## Where the tool stands

The core is solid and unusually modular for an Obsidian plugin. Registries decouple value types, entry kinds, cluster addons, derivations, section templates, layout presets and skill presets from the rendering shell; feature modules (rolling, dnd5e) register and migrate themselves; the influence engine turned every previously hard-coded game rule into user-editable building blocks; per-note state, multi-edit via proxy diffing, and a measurement-driven responsive layout round it out. The weakest areas are testability, the expressiveness ceiling of derivations and dice, persistence/portability of user configuration, and everything that happens *between* notes rather than within one.

## The expression ceiling

Derivations today are named building blocks (formula strings with one input) and influence lists are flat sums. Users will inevitably want `floor((STR + DEX) / 2) + max(PB, 2)`. The right move is a small expression engine: a tokenizer and Pratt parser producing an AST over numbers, property references (by name or short form), the existing building blocks as functions, and a fixed function library (floor, ceil, round, min, max, clamp, abs, if). Evaluation should run against a per-note dependency graph built once per render — topologically sorted, memoized, with cycles reported as a UI badge on the offending entry rather than silently depth-capped as today. The depth cap can remain as a backstop. Crucially the engine must stay a pure module (no Obsidian imports) so it is unit-testable, and the existing `mode`/`formula` influences should compile onto it so nothing breaks: a migration rewrites `{source, mode: "abilityMod"}` into the equivalent expression while the old fields keep working for one major version.

## Dice notation and the roll pipeline

The roller widget assembles chains structurally; power users will type them. Both should land on one representation: a roll AST (`2d6kh1 + 1d8 + DEX + 3`, keep/drop, exploding, reroll-below, success counting). Architect this as a dice expression parser in `utils/` mirroring the property expression engine, with `RollService.roll` accepting an AST instead of a single `DiceSpec` plus parts. The animation already handles labeled parts and dropped dice; it generalizes to one dice row per dice term. Advantage/disadvantage then becomes syntactic sugar for keep/drop, which is what v2.10 already implemented mechanically. On top of that fall out almost for free: saved roll macros (named ASTs in settings, surfaced in the roller and the right-click roll menu), per-entry custom roll formulas overriding the modifier sum, and configurable crit rules (crit range, which dice double) as settings consumed by the tone evaluation.

The roll log is currently per-view, capped at six entries, and erased on note switch. It should become a service-backed persistent history: an append-only array in settings (or a vault file, see persistence below) with timestamp, note, label, AST, faces, and total; the panel renders the tail; a command exports the session to a note. The dismiss-all/pin/copy affordances from v2.10 then operate over a real record rather than transient DOM.

## Rolling and properties inside notes

Everything lives in the sidebar; the note body is untouched. The single highest-leverage missing feature is inline rendering: a markdown post-processor that turns `` `roll: 2d6+DEX` `` into a clickable roll chip in reading mode, and `` `prop: Strength` `` into a live value (editable on click) resolved through the same NoteModel. Architecture: register one code-block processor and one inline post-processor in `main.ts`, both thin adapters that reuse the existing renderers and RollService — no new rendering logic. This also creates the path to statblocks-in-notes, which is what most TTRPG users ultimately want.

## Cross-note features

The engine is strictly single-note. Missing: aggregation ("party HP" summing a property across notes with a given type), references ("mount's Speed" reading another note's property via a link-typed property), and a table view of all notes of a type. These need an index service subscribing to `metadataCache` events, maintaining `type → notes → values`, with reactive invalidation; influences gain an optional `note` qualifier resolved through it. Keep it lazy — build the index only when a layout actually uses a cross-note source — because most vaults will never need it. The type table view is then a separate ItemView reusing the section renderer in a transposed mode.

## Validation, conditions, and richer types

Three structural gaps in the property model. Validation: min/max exist for numbers but nothing enforces them on entry, and strings/lists have nothing — add an optional constraint object per entry (range, regex, allowed values, required) checked in the inline editors with non-blocking error styling. Conditional visibility: entries and sections should accept a visibility expression ("show Spell Slots when Class is Wizard") evaluated by the same expression engine; this replaces the blunt hideIfEmpty. Types: units with display conversion (lb/kg, gp/sp), a duration/date-math type, and a link type that understands vault paths (rendering as a proper internal link and feeding cross-note references above).

## Persistence and portability

All layouts, types, derivations and abbreviations live in `data.json`, which is per-vault but invisible, unversionable, and unshareable. Three additions, in order of value: an export/import of any type or section template as a JSON snippet (one command, one modal — the registries already serialize cleanly); an optional vault-file mode where layouts are stored in a designated folder of JSON files so they sync, diff and share with the vault itself, with `data.json` retained for plugin-level settings only; and automatic timestamped backups of settings before every migration, because migrate() already rewrites user data on upgrade and one bug there is unrecoverable. Settings migrations themselves should move from ad-hoc per-module checks to an explicit versioned migration table (settings.schemaVersion, ordered steps, each pure and tested).

## Testing and code health

There are no tests, and the codebase has been refactored heavily across nine versions — the risk compounds. The pure modules (dice parsing/rolling, influences, migrations, expression engine when it lands) are already free of Obsidian imports or nearly so; add vitest with a thin obsidian stub (one already exists for the build) and write golden tests for migrations against captured real `data.json` samples, property-based tests for the dice math (sum of kept dice equals total minus modifier, drop semantics for adv/dis), and snapshot tests for denotation text. The `ext<T>()` casts are unchecked reads of persisted data; a lightweight runtime validator at load (per-field type coercion in `normalize`, which exists but is manual) is more proportionate than a schema library. The legacy record-based skills type and the stubbed dnd5e files should get a deprecation plan: one release that auto-prompts conversion, then removal. The responsive pass forces synchronous reflows per row; if large sheets ever feel sluggish, batch reads before writes (it is already close) and skip sections whose width signature has not changed since the last pass.

## UI and accessibility

Keyboard support is thin: the sidebar should be tabbable end-to-end with arrow-key movement between entries, Enter to edit, and visible focus rings; popups (dice menu, roll menu, section options) need focus traps and Escape consistently. Screen-reader support means real roles on the toggle checkboxes, labels tied to inputs, and aria-live on the roll result so a roll announces its total. The animation honors reduced motion, but the squeeze transitions should too (they partially do). On mobile, options modals would serve better as bottom sheets, and the 24px squeeze slack could scale with font size rather than being a fixed pixel count. Theming: expose the remaining hard-coded sizes and the accent usage as `--ep-*` CSS variables so Style Settings users can restyle without `!important` wars.

## Distribution

The plugin is at the polish level where community release is the natural next step. That requires: a real author/repo in `manifest.json`, a LICENSE file, a GitHub repository with CI (tsc, build, attach the release zip — the build is already two commands), BRAT compatibility for beta testers (which works as soon as releases carry `main.js`/`manifest.json`/`styles.css`), and submission to the community plugin directory. The i18n dictionaries should move from TS objects to JSON files loadable by locale so translators can contribute without touching code; the registry already merges per-module dictionaries, so only the loader changes.

## Constraints to respect going forward

Frontmatter is the single source of truth and other plugins write to it too: never persist derived values into notes unless a user explicitly opts in, debounce writes (rapid slider drags currently produce write bursts that sync tools amplify), and treat external file changes during an inline edit as a conflict to surface, not overwrite. Keep game-specific logic out of the core — the influence engine's genericity is the project's main asset, and every new feature should ask "is this a building block or a preset built from blocks?" Obsidian API drift is the main external risk (frontmatter processing and view lifecycle have both changed before); the thin adapter layer in NoteModel and the view is the right insulation, so resist letting Obsidian types leak deeper into the engine. Finally, the settings object is becoming the kitchen drawer of the project; before adding the next ten features, split it into namespaced sub-objects with the migration table above, while the registries keep doing what they already do well.
